# Data Export Spec

## Status

Future Settings dependency. This spec documents export behavior only. No export code, migration, or
Edge Function is implemented in this phase.

## Problem

Users need a clear way to take their Habit Compass data with them before account deletion or for
personal backup without exporting authentication credentials or internal system metadata.

## Scope

- Export as CSV ZIP.
- Export as JSON.
- Android save/share behavior.
- Export schema versioning.
- Authorization, RLS, security, and failure handling.

## Non-Goals

- Import/restore.
- Sync to third-party storage.
- Export of passwords, OAuth tokens, session data, account-security metadata, or user Settings
  preferences.
- Production implementation in this documentation phase.

## Product Behavior

The Data and privacy screen exposes two export actions:

- Export as CSV.
- Export as JSON.

Exports include active and archived app data owned by the authenticated user. Exports exclude user
Settings preferences and authentication/account information.

Suggested filenames:

- `habit-compass-export-YYYY-MM-DD.zip`
- `habit-compass-export-YYYY-MM-DD.json`

Use the user's current local date for the filename. Export payload timestamps use ISO 8601 strings.

## Generation Strategy

Production exports should be generated by a secure server-side path, preferably a Supabase Edge
Function, because it can:

- Authenticate the requesting user.
- Apply or deliberately mirror RLS-scoped user ownership.
- Aggregate normalized tables consistently.
- Avoid exposing privileged credentials.
- Handle larger exports and temporary-file cleanup.

Client-side generation is acceptable only for the mock/local development data source or as a thin
download wrapper around an authenticated server response. If temporary files are stored, use a
private Supabase Storage bucket with user-scoped paths, short-lived signed URLs, and scheduled
cleanup.

## Included Data

Include user-owned app data:

- Categories.
- Habits.
- Habit completion and skip logs.
- Habit inactivity periods.
- Tasks.
- Recurrent tasks.
- Recurrent task logs.
- Mood logs when present.
- Reflections when present.
- Weekly records, including existing `week_start` / `weekStartDate`, future `period_end` /
  `periodEnd` once implemented, weekly focus, Big Rocks, review answers, and nested
  mood/reflection references where stored.
- Archived records.

Include stable IDs and relationships so exported data can be understood without flattening unrelated
entities.

## Excluded Data

Exclude:

- Passwords.
- Authentication credentials.
- OAuth access or refresh tokens.
- Supabase session information.
- Internal account-security metadata.
- User Settings preferences such as language, theme, feature flags, and week start.
- Internal administrative fields that have no user value.
- Service-role, API, or backend operational information.

The user's authentication/account information is not part of the export.

## CSV ZIP Contract

CSV export produces a ZIP archive containing separate UTF-8 CSV files. Use `\n` line endings and a
single header row per file.

Minimum files:

- `categories.csv`
- `habits.csv`
- `habit_inactivity_periods.csv`
- `tasks.csv`
- `recurrent_tasks.csv`
- `recurrent_task_logs.csv`
- `completion_logs.csv`
- `mood_logs.csv`
- `reflections.csv`
- `weekly_records.csv`
- `weekly_big_rocks.csv`

If future normalized tables are added, add separate CSV files rather than flattening unrelated
entities into one oversized CSV.

### CSV Formatting

- Encoding: UTF-8.
- Header names: stable snake_case.
- Strings: RFC 4180-style quoting when a value contains comma, quote, CR, or LF.
- Quotes inside strings are doubled.
- Null values: empty cell.
- Booleans: `true` or `false`.
- Dates: `YYYY-MM-DD` local date strings.
- Timestamps: ISO 8601 UTC strings.
- Durations: integer minutes.
- Decimal quantities: plain decimal string with `.` separator.
- JSON config fields: compact JSON string inside the cell.

## JSON Contract

JSON export produces one complete structured export.

Top-level shape:

```json
{
  "schemaVersion": "1.0.0",
  "generatedAt": "2026-06-19T00:00:00.000Z",
  "app": {
    "name": "Habit Compass",
    "exportFormat": "json"
  },
  "data": {
    "categories": [],
    "habits": [],
    "habitInactivityPeriods": [],
    "tasks": [],
    "recurrentTasks": [],
    "recurrentTaskLogs": [],
    "habitLogs": [],
    "moodLogs": [],
    "reflections": [],
    "weeklyRecords": []
  }
}
```

Requirements:

- Include export schema version.
- Include generation timestamp.
- Group data by entity.
- Preserve stable IDs.
- Preserve relationships by ID.
- Preserve explicit dates and timestamps.
- Include archived records.
- Include weekly records with existing `weekStartDate` and future `periodEnd` once implemented.
- Include Big Rocks nested under weekly records or as a clearly related array.
- Include mood and reflections when stored or referenced by weekly records.
- Exclude Settings preferences and authentication/account data.

## Weekly Records In Exports

Weekly records must export explicit historical intervals. Do not derive historical weekly record
ranges from the user's current week-start preference during export.

Derived analytics may regroup completion logs by the current preference in the app, but export must
preserve saved weekly planning intervals exactly.

## Android Save And Share

- Android implementation should use the platform share/save flow once Capacitor native integration
  is approved.
- Web fallback should download through the browser.
- Development builds without a production package/listing should still export locally.
- Export success should not require email.

## Authorization And RLS

- Only authenticated users can export their own data.
- Export functions must fetch only the requesting user's rows.
- Server-side exports may use privileged credentials only inside trusted server/Edge Function code.
- Service-role keys must never be exposed to client code.
- RLS policies remain the baseline for client-accessible tables.
- Any private Storage temporary export object must use user-scoped paths and short-lived signed URLs.

## Failure Handling

Handle:

- Offline state.
- Expired session.
- Unauthorized request.
- Export timeout.
- Temporary Storage upload failure.
- Large export memory limits.
- ZIP generation failure.
- User cancellation of Android share/save.
- Cleanup failure.

Failures use localized generic messages with a support-safe error identifier. Do not expose SQL,
Supabase internals, or storage paths to users.

## Temporary File Cleanup

- Prefer streamed generation without persistence when practical.
- If persistence is needed, temporary exports live in a private bucket.
- Suggested retention: `[EXPORT TEMP FILE RETENTION TO CONFIRM]`.
- Cleanup should run through a server-side scheduled job or equivalent.
- Cleanup must be idempotent and safe to retry.

## Acceptance Criteria

- Given a user exports CSV, when generation succeeds, then a ZIP file with separate UTF-8 CSV files
  is produced.
- Given a user exports JSON, when generation succeeds, then one schema-versioned JSON file is
  produced.
- Given app data contains archived records, then the export includes those archived records.
- Given a weekly record was created under a previous week-start preference, then export preserves
  its stored `week_start` / `weekStartDate` and future `period_end` / `periodEnd`.
- Given user Settings preferences exist, then they are excluded from CSV and JSON exports.
- Given authentication/session data exists, then it is excluded from CSV and JSON exports.
- Given another user's data exists, then it is not present in the export.
- Given generation fails, then no privileged details are shown to the user.

## Test Plan

- Unit tests for CSV escaping, null handling, boolean formatting, date formatting, JSON shape, and
  schema versioning.
- Unit tests for weekly interval preservation.
- Integration tests for authenticated export authorization, RLS scoping, Edge Function responses,
  Storage temporary upload/download, and cleanup.
- E2E tests for Export as CSV, Export as JSON, Android/web fallback, cancellation, and error states.
- Security tests proving credentials, sessions, settings, and other users' rows are excluded.
